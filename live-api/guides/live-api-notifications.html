    Live-API: Benachrichtigungen

    In diesem Thema wird gezeigt, wie Benachrichtigungen für Live-API-Jobs eingerichtet werden.

    Einführung

    Das Brightcove Live-System sendet Benachrichtigungen über verschiedene Ereignisse. Sie können Listener für Benachrichtigungen einrichten, um weitere Aktionen auszulösen. Die Listener können in jeder serverseitigen Sprache geschrieben werden, die Sie verwenden, und sie würden nach eingehenden POST-Anfragen suchen, den JSON-Anfragetext analysieren und dann die gewünschten Maßnahmen ergreifen. Im Folgenden erfahren Sie, wie Sie Benachrichtigungen anfordern können.

    Benachrichtigungen anfordern

    Sie können Benachrichtigungen anfordern, indem Sie ein oder mehrere notifications Felder in die Ausgaben Ihrer Job-Create-Anfrage einbeziehen.

    Der Wert von notifications ist ein Array von Zielobjekten für Benachrichtigungen. Sie können eine einfache Zeichenfolge mit einer URL verwenden: "http://httpbin.org/post" oder Sie können ein Objekt mit folgenden Optionen verwenden: {"url": "http://httpbin.org/post", "credentials": "reference_to_your_credentials"}. Eine Benachrichtigung wird an das von Ihnen angegebene Ziel gesendet, wenn die Ausgabe ihren Status ändert. Wenn Sie diesen Parameter auf eine VOD-Ausgabe anwenden, beziehen sich die Benachrichtigungen nur auf diesen VOD-Job, nicht auf den Live-Job.

    Bitte beachten Sie. Nur eine Benachrichtigungs-URL pro Ereignistyp ( first_segment_uploaded , state_change usw.) ist erlaubt.

    Retry-Strategie

    Im Falle einer fehlgeschlagenen Anfrage zum Senden einer Benachrichtigung besteht die standardmäßige Wiederholungsstrategie darin, 50-mal mit einer exponentiellen Verzögerung zwischen den Versuchen erneut zu versuchen.

              max_retry_times = 50
          delay_delta_s = 5
          next_retry = now_s + retry_count * delay_delta_s

    Veranstaltungen

    Im Folgenden finden Sie die Ereignisse eines Job-Lebenszyklus, den Sie anhören können. Beachten Sie, dass der Ereignisbereich pro Job gilt, sodass der Ereignislebenszyklus für einen Live-Job vom Lebenszyklus für einen VOD-Clipping-Job, der auf demselben Live-Stream basiert, getrennt ist.

    • state_changed - der Status des Live-Jobs wurde geändert; Einzelheiten finden Sie in der Tabelle unten
    • first_segment_uploaded - das erste Segment des Livestreams wird zum Ursprung hochgeladen
    • output_finished - Das Live-Event ist beendet, und die VOD-Verarbeitung hat mindestens eine Audio- und eine Videowiedergabe oder eine progressive MP4-Wiedergabe erstellt

    Der state_changed wird über die folgenden in der folgenden Tabelle beschriebenen Zustände informiert

    disconnected Der Encoder wurde getrennt und der Streaming-Worker wartet auf eine Wiederverbindung.

    processing Der Clip wird bearbeitet.
    creating_asset Ein VOD-Asset wird erstellt.
    cancelling Der Auftrag wurde abgebrochen und der Worker stoppt den Job und verarbeitet KEINE ausstehenden VOD-Ausgaben.
    cancelled Der Job wurde erfolgreich abgesagt.
    finished Der Job ist erfolgreich abgeschlossen.
    failed Der Auftrag wurde aufgrund eines Systemfehlers beendet.
    SSAI-Staaten (in Mitteilungen als gemeldet ssai_state)
    none Der Job ist kein SSAI-Job.
    waiting_input Der Streaming-Worker wartet darauf, dass sich der Encoder verbindet und die Informationen zur Stream-Eingabe bereitstellt.
    start_transcoding Der Encoder wurde angeschlossen, und die SSAI-Slate-Seite wurde für die Transkodierung basierend auf den Stream-Eingangsdaten in die Warteschlange gestellt und Daten ausgegeben.
    transcoding Der Schiefer wird transcodiert.
    error Die Tafel konnte nicht heruntergeladen oder transcodiert werden.
    ready Die Tafel wurde generiert und der Auftrag ist bereit für die SSAI-Wiedergabe.
    SEP States (berichtet in Mitteilungen als sep_state)
    none Der Job ist kein SEP-Job.
    ready Der Einstiegspunkt wurde aktiviert und ist bereit für die Verbindung des Encoders.
    pending_activation Der Einstiegspunkt wurde für die Streaming-Arbeitnehmerzuweisung in die Warteschlange gestellt.
    activation_in_progress Der Einstiegspunkt wird aktualisiert, um Verbindungen zum Streaming-Worker weiterzuleiten.
    pending_deactivation Der Einstiegspunkt wurde in die Warteschlange für die Bereinigung vom Streaming-Worker gestellt.
    deactivation_in_progress Der Einstiegspunkt wird vom Streaming-Worker getrennt.
    cancelled Der Einstiegspunkt wurde annulliert.
    finished Der Einstiegspunkt wurde erfolgreich abgeschlossen.
    RTMP-Ausgangszustände (für rtmp_output_state_changed Benachrichtigungen als gemeldet state)
    starting Der Stream beginnt.
    connected Der Encoder hat eine Verbindung hergestellt.
    Disconnected Der Encoder hat die Verbindung getrennt.
    error Der Stream konnte keine Verbindung herstellen.
    pending_deactivation Der Einstiegspunkt wurde in die Warteschlange für die Bereinigung vom Streaming-Worker gestellt.
    deactivation_in_progress Der Einstiegspunkt wird vom Streaming-Worker getrennt.
    cancelled Der Einstiegspunkt wurde annulliert.
    finished Der Einstiegspunkt wurde erfolgreich abgeschlossen.

    Hinweise

    • In bestimmten Fällen werden identische Benachrichtigungen mehr als einmal gesendet. Wenn Ihr Handler basierend auf Benachrichtigungen Aktionen (abgesehen von der einfachen Protokollierung) ergreift, sollten Sie ihn so konfigurieren, dass er nach Duplikaten (mehrere Benachrichtigungen mit demselben id und status ) prüft und diese ignoriert.

    Richten Sie Benachrichtigungen im Live-Modul ein

    Um Benachrichtigungen für ein Live-Ereignis einzurichten, das im Live-Modul erstellt wurde, müssen Sie Folgendes tun:

    1. Erstellen Sie eine Handler-Anwendung, die POST-Anfragen wie die im vorherigen Abschnitt gezeigte PHP-App erhalten kann.
    2. Hosten Sie die App auf einer öffentlichen URL.
    3. Wenn Sie Ihren Live-Job im Live-Modul erstellen, erweitern Sie die erweiterten Optionen.
