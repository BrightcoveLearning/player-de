    Live-API: Cue-Points und Ad Beacons mit SSAI

    In diesem Thema erfahren Sie, wie Sie Cue-Points und Ad Beacons verwalten, wenn Sie Brightcoves Server-Side Ad Insertion (SSAI) für Ihre Live-Stream-Jobs verwenden.

    Übersicht

    Serverseitige Anzeigeneinfügung (SSAI) ermöglicht es Ihnen, Anzeigen während eines Live-Streaming-Events zu bestimmten Zeiten anzuzeigen. Allgemeine Informationen finden Sie in der Live-API: Serverseitiges Einfügen von Anzeigen (SSAI) Dokument.

    Bevor Sie es verwenden können SSAI, muss es für Ihr Konto aktiviert sein. Ihr Kontomanager teilt Ihnen gerne weitere Details hierzu mit.

    Cue-Points

    Werbeblöcke werden durch Cue-Points ausgelöst, die auf zwei Arten angegeben werden können:

    • Vom Encoder an Brightcove gesendet
    • Sofortige Cue-Points, die über Live API

    Vom Encoder

    Das Live-Bereitstellungssystem von Brightcove kann vom Encoder eingereichte Cue-Points im AMF-Format interpretieren:

      AMFDataList
  [0]:onCuePoint
  [1]:{Obj[]:
    time: 1.9889, //Difference from PTS of THIS packet to the first PTS of the 1st video frame in the adbreak
    name: "scte35",
    type: "event",
    ad_server_data: "YWJjZGVmZ2g=",	// optional introduced by Brightcove. It is a base64 encoded json map of parameters e.g. {‘key’:’value’}
    parameters: {Obj[]:
      type: "avail_in",
      duration: 12.0
    }
  }

    Hinweise:

    • Derzeit werden nur avail_in Typ-Cue-Points unterstützt.
    • SCTE-35 Cue-Points werden für RTMP- und TS-Eingänge unterstützt.
    Im Anhang unten finden Sie einen Screenshot eines Beispiel-Cue-Point-Setups für den Elemental-Encoder.

    Manueller Cue-Point-Ein

    Sie können sofortige Cue-Points erstellen, Live API indem Sie eine POST Anfrage senden:

    -Methode POST
    URL https://api.bcovlive.io/v1/jobs/Job_ID/cuepoint
    Überschrift X-API-KEY: your API KEY

    Schließen Sie einen Anfragetext ein, der Folgendes angibt:

    Feld Art Beschreibung
    duration Ganzzahl Dauer der Pause in Sekunden.

    Die Dauer des eingefügten Cue-Points muss mindestens das Doppelte der Länge der Segmente im Auftrag betragen. Siehe das Beispiel für die Dauer.
    timecode SMPTE-Formate OPTIONAL: Ein Timecode im SMPTE-Format, HH:MM:SS:FF (FF = Frames), um anzugeben, wann ein Satz beliebiger Variablen (Schlüssel/Wert-Paare) an den AdServer übergeben werden soll.

    Wenn nicht angegeben, wird der Cue-Point sofort eingefügt.

    Wenn Sie die Eigenschaft timecode verwenden, muss der Encoder SMPTE-formatierten (HH:MM:SS:FF) -Timecode senden, der in der tc Eigenschaft über OnFI. Timecodes sind vom Beginn des Livestreams an.
    ad_server_data Objekt OPTIONAL: Die Schlüssel/Wert-Paare, die Sie übergeben, hängen vom verwendeten Ad-Server ab. Weitere Informationen finden Sie in der Dokumentation zu Ihrem Ad-Server und im Abschnitt Targeting-Anzeigen mit Anzeigenmakros .

    Beispiel für die Dauer

    Die Dauer des eingefügten Cue-Points muss mindestens das Doppelte der Länge der Segmente im Auftrag betragen.

    Das Einfügen eines 10-Sekunden-Cue-Points in einen Job mit funktioniert "segment_seconds"=4 beispielsweise einwandfrei. Das Einfügen desselben Cue-Points in einen Job mit "segment_seconds"=6 führt jedoch zu folgendem Fehler:

      "error": "The parameter duration should be greater than
    or equal to (2 * target duration) of the job"
     

    Muster-Anfrage

      {
    "duration": 30,
    "timecode": "15:50:49:16",
    "ad_server_data" : {
    "adbreakid": 12312
    "breaktheme": "fitness"
    }
  }

    Hinweise

    1. Software-Encoder wie Wirecast und OBS unterstützen den sendenden Timecode über OnFI Pakete im RTMP-Stream nicht
    2. Elementare Hardware-Encoder unterstützen den sendenden Timecode über OnFI Pakete im RTMP-Stream

    Sample-Antwort

      {
    "id": "Job_ID",
    "cue_point": {
      "id": "adBreak-2f58393ada1442d98eca0817fa565ba4",
      "duration": 30,
      "accuracy": "segment", [Can be segment or frame  ]
      "inserted_at": "2017-07-21T09:30:46.307Z" [ Time when the cue point was inserted in the stream]
    },
  }

    Beacons

    Beacons sind Datenpunkte bei der Wiedergabe, die an Analysen von Drittanbietern gesendet werden, um zu verfolgen, ob und wie viele Anzeigen gespielt wurden. In diesem Abschnitt werden wir uns die Beacon-Typen ansehen, die mit dem festgelegt werden können Live API, und Variablen, die zur Bereitstellung der Daten verwendet werden können. Im nächsten Abschnitt werden die API-Anfragen beschrieben, die zum Erstellen und Verwalten von Beacon-Sets verwendet werden.

    Arten von Beacon

    Beacon-Typen
    Beacon-Typ Beschreibung
    Load Wird einmal pro Sitzung ausgelöst und nur ausgelöst, wenn das Top-Level-Manifest angefordert wird
    Play Inhalt wurde angefordert und das erste Segment zurückgegeben
    Heartbeat Zieldauer (Segmentsekunden)
    AdStart Einzelanzeige wurde gestartet
    AdFirstQuartile Erstes Ad-Quartil (25%)
    AdMidpoint Zweites Ad-Quartil (50%)
    AdThirdQuartile Drittes Ad-Quartil (75%)
    AdComplete Einzelne Anzeige abgeschlossen
    AdBreakStart Werbeblock hat begonnen
    AdBreakComplete Werbeblock ist beendet

    Beacon-Ad-Variablen

    Die folgende Tabelle zeigt die Variablen, mit denen Sie Daten für die Beacon-URLs bereitstellen können. Um eine Variable einzubeziehen, umgeben Sie mit doppelten geschweiften Klammern wie folgt: {{job.job_id}} . Im nächsten Abschnitt zur Verwaltung von Beacon-Sets finden Sie vollständige Beispiele.

    Ad-Konfigurationsvariablen
    Variabel
    Beschreibung
    session.session_id
    eindeutige Sitzungskennung
    job.job_id
    eindeutige Job-ID
    videocloud.video_id
    Nur verfügbar für Jobs, die mit einem VideoCloud-Video erstellt wurden.
    application_ad_configuration.description
    Wert der Anwendung bei der Sitzungserstellung
    random.int32
    zufällige 32-Bit-Ganzzahl mit Vorzeichen
    random.int64
    zufällige 64-Bit-Ganzzahl mit Vorzeichen
    random.uint32
    zufällige 32-Bit-Ganzzahl ohne Vorzeichen
    random.uint64
    zufällige 64-Bit-Ganzzahl ohne Vorzeichen
    random.uuid
    zufällige uuid
    server.timestamputc
    Epochenzeit in Millisekunden, wenn der Aufruf von der ads-api getätigt wurde
    client.useragent
    HTTP-User-Agent-Header-Wert bei Sitzungser
    client.ipaddress
    http x-forwarded-for header value bei der Sitzungserstellung, falls angegeben, sonst die Remote-Adresse
    client.referrer
    http Referer Header-Wert bei der Sitzungserstellung (Hinweis: das ist die richtige Schreibweise)
    client.referer
    HTTP-Referer-Header-Wert bei der Sitzungserstellung (HTTP-Rechtschreibung)
    client.ipuaid
    Hash-Wert von client.ipaddress und client.useragent
    live.adbreak
    (derzeit unbenutzt)
    live.adbreakdurationms
    Dauer der Werbeunterbrechung in Millisekunden
    live.adbreakduration
    Dauer des Werbeblumens in Gleitkomma-Sekunden mit doppelter Genauigkeit
    live.adbreakdurationint
    Dauer des Anzeigenumbruchs in ganzzah
    live.adbreak.timestamputc.wallclock
    Epochenzeit in Millisekunden, wenn der Aufruf an den Ads-Server getätigt wurde
    live.adbreak.timestamputc.origin
    Epochenzeit in Millisekunden aus der Ursprungschunklist. Dieser Wert gibt den Zeitpunkt an, zu dem der Cue-Out-Chunk in der Ursprungschonliste erstellt wurde.
    live.adbreak.timestamputc.session
    Epochenzeit in Millisekunden aus der Ssai-Chunklist. Dieser Wert gibt die Zeit des Cue-Out-Chunk in der Ssai-Chunklist an. Da der Adbreak-Inhalt und die Adbreak-Lücke in der Regel NICHT gleich sind, unterscheidet live.adbreak.timestamputc.origin sich der nach dem 1. Adbreak von live.adbreak.timestamputc.session . Dieser Wert berücksichtigt die Zeitanpassungen, die in der SSAI Chunkliste vorgenommen wurden.

    SCTE-35 Anzeigenvariablen

    Die Society of Cable Telecommunications Engineers (SCTE) definiert einen Standard für die dynamische Anzeigeneinfügung von Livestreams. In der folgenden Tabelle werden die Konfigurationsvariablen für SCTE-35-Anzeigen zusammengefasst.

    Die Anzeigenparameter SCTE-35 werden derzeit sowohl mit RTMP- als auch von Transport Stream (TS) -Eingangsquellen unterstützt, die das Real-Time Transport Protocol (RTP) mit oder ohne Forward Error Correction (FEC) verwenden. Beachten Sie, dass FEC für RTP dringend empfohlen wird.

    Diese Anzeigenparameter sind bei Eingabestreams mit dem Real-Time Messaging Protocol (RTMP) nicht verfügbar.
    SCTE-35 Ad-Konfigurationsvariablen
    Variabel
    Beschreibung
    scte35_eventID
    eine eindeutige Ereignis-ID wurde mit einer SCTE35-Nachricht übergeben.
    scte35_programID
    Eine eindeutige Programm-ID wurde mit einer SCTE35-Nachricht übergeben.
    scte35_availNum
    Eine ID für eine bestimmte Spleißzeit, die für Anzeigen verfügbar ist, senden Sie über eine SCTE35-Nachricht.
    scte35_breakDuration
    Die Unterbrechungsdauer für die Werbeunterbrechung in Bezug auf die Ticks der 90-kHz-Takt des Programms wurde mit einer SCTE35-Nachricht bestanden.
    scte35_spliceTime
    Die Spleißzeit für eine Werbeunterbrechung in Bezug auf die Ticks der 90-kHz-Takt des Programms wurde mit einer SCTE35-Nachricht überschritten.

    Verwalten von Beacon-Sets

    Dieser Abschnitt enthält Details zu den API-Anfragen zur Verwaltung von Beacon-Sets. Im vorherigen Abschnitt finden Sie Beacon-Typen und Variablen.

    Um einem Live-Job ein Beacon-Set hinzuzufügen, erstellen Sie zuerst das Beacon-Set und fügen Sie dann die ID hinzu, wenn Sie den Job erstellen, wie folgt:

      {
  "live_stream": true,
  "region": "us-west-2",
  "reconnect_time": 30,
  "ad_insertion": true,
  "beacon_set": "beacon_set_id", ...

    Erstellen Sie ein Beacon-Set

    Um ein Beacon-Set zu erstellen, senden Sie eine POST Anfrage:

    -Methode POST
    URL https://api.bcovlive.io/v1/ssai/beaconsets
    Überschrift X-API-KEY: your API KEY

    Muster-Anfrage

      {
    "account_id": "User's Account ID [Optional]",
    "beacon_urls": [
      {
        "beacon_url": "https://myserver.com/beaconRX/{{job.job_id}}/load?position=load&sid={{session.session_id}}&jid={{job.job_id}}&app={{application_ad_configuration.description}}&rnd32={{random.int32}}&rnd64={{random.int64}}&bid={{random.uuid}}&t={{server.timestamputc}}&ua={{client.useragent}}&ip={{client.ipaddress}}&ref={{client.referrer}}&ref={{client.referer}}&ab={{live.adbreak}}&abd={{live.adbreakduration}}&abdi={{live.adbreakdurationint}}",
        "beacon_type": "Load"
      },
      {
        "beacon_url": "https://myserver.com/beaconRX/{{job.job_id}}/play?position=play&sid={{session.session_id}}&jid={{job.job_id}}&app={{application_ad_configuration.description}}&rnd32={{random.int32}}&rnd64={{random.int64}}&bid={{random.uuid}}&t={{server.timestamputc}}&ua={{client.useragent}}&ip={{client.ipaddress}}&ref={{client.referrer}}&ref={{client.referer}}&ab={{live.adbreak}}&abd={{live.adbreakduration}}&abdi={{live.adbreakdurationint}}",
        "beacon_type": "Play"
      }
    ]
  }
    Das account_id Feld ist die Live-Konto-ID. Wenn nicht angegeben, wird die Konto-ID des anfragenden Benutzers verwendet.

    Sample-Antwort

      {
    "beacon_set": {
      "beacon_urls": [{
      "beacon_url": "https://myserver.com/beaconRX/{{job.job_id}}/load?position=load&sid={{session.session_id}}&jid={{job.job_id}}&app={{application_ad_configuration.description}}&rnd32={{random.int32}}&rnd64={{random.int64}}&bid={{random.uuid}}&t={{server.timestamputc}}&ua={{client.useragent}}&ip={{client.ipaddress}}&ref={{client.referrer}}&ref={{client.referer}}&ab={{live.adbreak}}&abd={{live.adbreakduration}}&abdi={{live.adbreakdurationint}}",
      "beacon_type": "Load"
    },
    {
      "beacon_url": "https://myserver.com/beaconRX/{{job.job_id}}/play?position=play&sid={{session.session_id}}&jid={{job.job_id}}&app={{application_ad_configuration.description}}&rnd32={{random.int32}}&rnd64={{random.int64}}&bid={{random.uuid}}&t={{server.timestamputc}}&ua={{client.useragent}}&ip={{client.ipaddress}}&ref={{client.referrer}}&ref={{client.referer}}&ab={{live.adbreak}}&abd={{live.adbreakduration}}&abdi={{live.adbreakdurationint}}",
      "beacon_type": "Play"
    }],
    "beacon_set_id": "Inserted Beacon Set ID",
    "account_id": "USER's ACCOUNT ID"
    }
    "inserted": true
  }

    Aktualisieren eines Beacon-Sets

    Das Aktualisieren eines Beacon-Sets ähnelt dem Erstellen eines Beacon-Sets. Senden Sie eine PUT Anfrage:

    -Methode PUT
    URL https://api.bcovlive.io/v1/ssai/beaconsets/beaconset/beacon_set_id
    Überschrift X-API-KEY: your API KEY

    Muster-Anfrage

      {
    "account_id": "User's Account ID [Optional]",
    "beacon_urls": [
      {
      "beacon_url": "https://myserver.com/beaconRX/load",
      "beacon_type": "Load"
      },
      {
      "beacon_url": "https://myserver.com/beaconRX/play",
      "beacon_type": "Play"
      }
    ]
  }
    Das account_id Feld ist die Live-Konto-ID. Wenn nicht angegeben, wird die Konto-ID des anfragenden Benutzers verwendet.

    Sample-Antwort

      {
    "beacon_set": {
      "account_id": "User's Account ID",
      "beacon_set_id": "Beacon set ID",
      "beacon_urls": [{
        "beacon_url": "https://myserver.com/beaconRX/load",
        "beacon_type": "Load"
        },
        {
        "beacon_url": "https://myserver.com/beaconRX/play",
        "beacon_type": "Play"
      }],
      "updated_beacon_set": {
        "beacon_set_id": "Beacon set ID",
        "beacon_urls": [{
          "beacon_url": "https://myserver.com/beaconRX/load",
          "beacon_type": "Load"
        },
        {
          "beacon_url": "https://myserver.com/beaconRX/play",
          "beacon_type": "Play"

    Holen Sie sich Beacon-Sets

    Um die Beacon-Sets für ein Konto abzurufen, senden Sie eine GET Anfrage:

    -Methode GET
    URL https://api.bcovlive.io/v1/ssai/beaconsets/account/Account ID
    Überschrift X-API-KEY: your API KEY
    Beachten Sie, dass die Konto-ID hier die Live-Konto-ID ist.

      "account_id": "User's Account ID",
      "beacon_set_id": "Beacon set ID1",

      "beacon_type": "Load"
    }]
    },

    Holen Sie sich ein Beacon nach ID

    Um einen einzelnen Beaconsatz durch seine ID abzurufen, senden Sie eine GET Anfrage:

    -Methode GET
    URL https://api.bcovlive.io/v1/ssai/beaconsets/beaconset/beacon_set_id
    Überschrift X-API-KEY: your API KEY

    Sample-Antwort

      {
      "account_id": "User account ID",
      "beacon_set_id": "Beacon set ID",
      "beacon_urls": [{
        "beacon_type": "Load",
        "beacon_url": "https://myserver.com/beaconRX2/load"
    },
    {
      "beacon_type": "Play",
      "beacon_url": "https://myserver.com/beaconRX2/play"
    }]
  }

    Löschen eines Beacon-Sets

    Um ein Beacon-Set zu löschen, senden Sie eine DELETE Anfrage:

    -Methode DELETE
    URL https://api.bcovlive.io/v1/ssai/beaconsets/beaconset/beacon_set_id
    Überschrift X-API-KEY: your API KEY

    Sample-Antwort

    Die Antwort wird so aussehen:

      {
    "beacon_set_id": "Beacon set ID",
    "deleted": true
  }
